## Task

Debug and fix race condition leaving empty grid cells after cascading matches.

## Root Cause

`CompressColumn` updated `_gameBoard[row, column]` but did not call `UpdateCoordinates()` on the tile being shifted. This left the tile's `.Row` and `.Column` properties **stale**. When cascade behaviors (e.g., `MatchDirectionEliminationBehavior`) read those stale coordinates to call `NullifyTileAt()`, they nullified the **wrong board cell**. The intended cell retained a recycling-marked tile; `DoRecycle()` recycled it in-place in `EndTurn()`, leaving a phantom empty cell.

## Fixes Applied

1. **CompressColumn sync (line ~1115):** Added `higherTile.UpdateCoordinates(row, column)` immediately after each `_gameBoard` swap, keeping tile position properties synchronized with the board.

2. **HandleMatches recycling (line ~975):** Changed to use board-authoritative positions from `MatchedTileInfo.tile.Row`/`.Column` instead of the tile object's potentially-stale properties.

3. **EndTurn drain (line ~end):** Added drain loops for `_moveRequests` and `_movingTiles` to prevent stale animation items from accumulating across turns.

## Impact

- Any code reading `tile.Row`/`tile.Column` after `CompressColumn` now sees correct values  
- Eliminates the null-cell-after-cascade bug  
- `EndTurn()` drain prevents stale animation items from bleeding between turns  
- No behavioral change for non-cascade flows
